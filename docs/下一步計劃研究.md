## 為什麼我會這樣判斷

你的 repo 現在其實已經不是普通 MAPF demo。README 裡明確說它是多機 AGV lifelong path-planning / traffic-control engine，核心目標是讓機器人在共享路網上 A→B 且不碰撞，並且已經有 in-memory closed-loop simulator 和 web visualizer。 架構上也已經切成 Map、PathPlanning、TrafficControl、Deadlock、Coordination、Simulation、Web 幾個 bounded contexts，不是單一演算法倉庫。

目前 v0 / v1 已經完成，v1 有 SIPP，v2 是 RHCR + 避免式預防，v3 目標是 CBS/CCBS + PIBT + SIPPwRT，換句話說，**v3 做完後你會已經有一套很完整的 solver stack**：優先級規劃、局部聯合規劃、連續時間、運動學雛形都碰到了。 這時候繼續做 v4 solver，邊際收益會下降，因為 Jiaoyang Li 這條線上已經有 RHCR、EECBS、MAPF-LNS2、CBSH2-RTC 這些成熟公開 solver。Jiaoyang 的 RHCR 是 lifelong MAPF，週期性呼叫 windowed MAPF solver，參數裡已經有 KIVA、SORTING、simulation window、planning window、WHCA/ECBS/PBS 等基準場景與 solver。  EECBS 是 bounded-suboptimal CBS，結合 EES、高層 focal / 低層 focal search、多種 CBS improvements，還有 SIPP low-level 選項。 MAPF-LNS2 則是 large neighborhood search 的 fast repairing / anytime 方向。

所以，**你後面真正稀缺的不是「我也有 CBS / PIBT / SIPP」；而是「我能把它包成真實 FMS 場景下可測、可調、可接入、可運營的交通控制平台」**。

## 我建議的 v4：SwarmRoute Lab + Traffic Intelligence

我會把 v4 定義成兩個綁在一起的東西：

> **SwarmRoute Lab**：真實倉儲 / 工廠車隊級 benchmark、scenario、metrics、trace、dashboard、planner 對比平台。
> **Traffic Intelligence**：基於 trace 和 congestion 的 guidance graph / zone capacity / edge weight / one-way-lane / dispatch policy 優化層。

這比「再加一個 solver」更值錢，原因是：最新研究方向已經明顯從純 MAPF solver 走向 **lifelong、realistic execution、kinodynamics、uncertainty、guidance graph、task assignment、large-scale evaluation**。Jiaoyang Li 目前在 CMU Robotics Institute，研究重點是 large teams of mobile robots 的協調，近期論文方向包含 windowed MAPF completeness、differential-drive 多機 motion planning、temporal plan graph、online guidance graph optimization、LNS2+RL、SMART/LSMART 等。([Jiaoyang Li][1])

尤其是 LSMART / SMART 這條線很值得你對齊。LSMART 明確把問題擴展到 FMS/AGV 的 lifelong MAPF 評測，指出真實系統不只有 planner，還包括 invocation policy、instance generator、planner、fail policy，並且加入 differential-drive kinodynamics、通信延遲、執行不確定性與 Action Dependency Graph。([GitHub][2]) 它還展示了 1000 robots warehouse 場景的模擬能力。([Smart Mapf][3])

## v4 最該做的 5 個模組，按價值排序

### 1. Benchmark / Scenario / Metrics 平台：先把「好不好」量化

這是第一優先級。你現在有 closed-loop simulator、web replay、collision-free verification，但下一步應該變成一個標準化實驗平台：同一張圖、同一批訂單、同一 seed，可以跑 v1/v2/v3、不同 planner portfolio、不同 dispatch policy，最後輸出可比較報告。

你現在 Simulation + Coordination 已經很適合做這件事。Coordination 每 tick 做 Map→ReservationView→PathPlanning→TrafficControl→prune-and-replan，而且順序、優先級、retry 都是 deterministic。 Simulation 的 executor 則負責 tick、right-of-way gate、release、parked-cell blockedResources、DidNotConverge 等閉環結果。

v4 應該補這些 metrics：

| 指標                             | 為什麼重要                |
| ------------------------------ | -------------------- |
| Throughput：任務/小時、完成訂單數         | 業界最關心                |
| Mean / P95 / P99 travel time   | 不能只看平均               |
| Planning latency P50/P95/P99   | solver 能不能在線上跑       |
| Replan attempts / agent        | 衡量交通壅塞與規劃穩定性         |
| Wait ratio / blocking time     | 衡量路網設計與交管品質          |
| Deadlock / DidNotConverge rate | 可靠性                  |
| Reservation conflict heatmap   | 找瓶頸點、瓶頸 lane、瓶頸 zone |
| Fairness / starvation          | 高優先級不能把低優先級餓死        |
| Energy proxy：距離、停走次數、轉向次數      | AGV/AMR 實際成本         |
| CPU / memory / lock contention | 進入 1000+ 車隊前必看       |

這件事做完，你的 SwarmRoute 就不是「看起來能跑」，而是「可以拿數據證明哪個 policy / solver / map design 更好」。

---

### 2. Guidance Graph / Traffic Engineering 層：這是 v4 的核心差異化

做完 v3 後，下一個高 ROI 方向是：**不要只讓 planner 在既有圖上找路，而是讓系統學會改變路網的交通規則**。

具體包括：

| 能力                            | 例子                                           |
| ----------------------------- | -------------------------------------------- |
| 動態 edge weight                | 某條 lane 最近太塞，成本升高                            |
| one-way / reversible lane     | 高峰期把某些走廊變成單向                                 |
| zone capacity                 | 某區最多同時進 N 台                                  |
| hotspot bypass                | 熱點 CP 周圍自動偏流                                 |
| station ingress/egress policy | 工作站入口排隊、出口讓路                                 |
| time-of-day traffic profile   | 不同任務流下使用不同權重                                 |
| offline map tuning            | 用 simulation 自動找更好 edge weights / directions |

這跟 Jiaoyang / ARCS 近期方向非常吻合。GGO / MGGO 這條線就是在優化 guidance graph；MGGO 明確是同時優化 edge directions 和 weights，Online GGO 則根據 real-time traffic patterns 產生動態 guidance，並改善 PIBT 等 lifelong MAPF 演算法的 throughput。([ARCS Lab][4]) ([ARCS Lab][4])

更關鍵的是，這個方向非常適合你的架構。Map 擁有 topology，TrafficControl 擁有 reservation / occupancy，Coordination 擁有每 tick 的車隊決策，Simulation 能產生 trace。你只要新增一層：

```text
Telemetry / Trace
        ↓
Congestion Analyzer
        ↓
GuidanceGraph Optimizer
        ↓
Map edge weights / lane direction / zone capacity / planner cost model
        ↓
PathPlanning + TrafficControl
```

這比直接硬塞更多 solver 更漂亮，因為它是**系統級優化**，不是單點演算法優化。

---

### 3. 真正的 Dispatcher / Task Assignment：從「路徑規劃」升級成「車隊管理」

業界真正在意的不是「某兩台車避開了」，而是「今天整個倉庫多出多少吞吐」。所以 v4/v5 必須補一個真正的 Dispatcher / Order domain。

你現在 Coordination 已經像 OpenTCS 的 Dispatcher / RHCR loop，架構設計也明確把 Map、PathPlanning、TrafficControl、Deadlock 分開，並參考 OpenTCS 的 Router / Scheduler / Dispatcher 分工。 但目前主體還是「agent goal 已給定後怎麼走」。下一階段應該做「任務怎麼派」。

建議新增：

```text
Order / Dispatch Context
- Task：pickup, dropoff, charge, park, buffer, return
- Station：工作站、貨架、充電站、暫停點
- Assignment：哪台車接哪個任務
- Queueing：站點入口排隊
- Battery / maintenance constraints
- SLA / priority / deadline
- Rebalancing：空車去哪裡等
```

這會讓 SwarmRoute 從 MAPF engine 變成 FMS core。Amazon Robotics 的方向也不是只做路徑，而是大規模倉儲 fleet 的 coordination / planning。DeepFleet 這類工作用大量機器人移動數據訓練模型來做 coordination and planning，說明業界價值在 fleet movement data + traffic management，而不只是單次路徑搜索。([arXiv][5])

---

### 4. Robust Execution：用 ADG / TPG 處理延遲、掉速、卡住

v3 如果包含 SIPPwRT / continuous-time，下一步一定會撞到「計畫時間 ≠ 真實執行時間」這個問題。你目前設計裡已經很重視 tick clock，因為 simulation 用 ManualFleetClock 把 reservation time axis 和 execution tick axis 對齊，避免 wall-clock 下 reservation 和實際執行脫鉤。 PathPlanning README 也記錄了 tick-vs-wall-clock 的問題。

但真實機器人會有：

* 加減速誤差
* 轉彎延遲
* 通信延遲
* 某台車慢了
* 某台車 emergency stop
* 某個 station 被人工佔用
* 地圖局部臨時封閉

所以 v4/v5 要做 robust execution layer。方向可以是：

```text
SpaceTimePath
    ↓
Action Dependency Graph / Temporal Plan Graph
    ↓
Executor follows dependencies, not exact timestamps only
    ↓
Delay propagates safely; no unnecessary global replan
```

Jiaoyang / ARCS 近期也很重視這點。SMART/LSMART 用 Action Dependency Graph 配合 kinodynamics 和 execution uncertainties；BTPG / TPG 方向則是在執行延遲下維持 robustness，處理 delays 造成的 inefficiencies、deadlocks、collisions。([GitHub][2]) ([ARCS Lab][4])

這對你的項目很關鍵，因為你的 collision-freedom 現在有兩層：reservation layer + executor right-of-way gate。 Robust execution layer 可以把它升級成「延遲也可控」，這才像真實 FMS。

做完 v3 後，你可以做 solver 對比與 portfolio，但我不建議第一步就 clean-room 重寫所有 Jiaoyang solver。更實際的是做 adapter：

注意一點：Jiaoyang 的 RHCR、EECBS、MAPF-LNS2 repo 都是 USC Research License，直接嵌入商用產品前要看 license；adapter / benchmark 對比會比直接 copy code 更安全。

這樣做的好處是：你可以讓 SwarmRoute 變成「MAPF / FMS 演算法實驗台」，而不是只跟自己比。這會直接提高項目的可信度。

## 不建議 v3 後立刻做什麼

### 不建議立刻 all-in 深度學習 / foundation model

不是說不要做，而是**不要第一個做**。SILLM、DeepFleet、RAILGUN 這些方向很強，但它們吃數據、吃訓練、吃基準場景，而且需要你先有高質量 trace。SILLM 已經做到 10,000-agent lifelong MAPF，並用 imitation learning + communication + global guidance 提升 throughput；DeepFleet 則是用 Amazon 倉儲裡大量 fleet movement data 來訓練 multi-agent foundation model。([ARCS Lab][4]) ([arXiv][5])

但另一邊，ARCS 對 LNS 的 reevaluation 也指出 rule-based heuristics 仍是很強的 baseline，目前 learning-based 方法不一定有清楚的 time / quality 優勢。([ARCS Lab][4]) 所以你的順序應該是：

```text
先做 trace / benchmark / guidance optimizer
再做 learning-assisted guidance / dispatch / neighborhood selection
最後才做 end-to-end learned policy
```

也就是：**先把數據飛輪做起來，再談 learning。**

## 你現在的核心抽象要升級

你目前 SwarmRoute 的架構其實已經打得很好：Map 管拓撲，PathPlanning 管單機規劃，TrafficControl 管時空預約，Coordination 做 RHCR loop，Simulation 做閉環執行。 目前 SIPP 是在 reservation view 上搜尋 safe interval，並把等待表現成 CP dwell interval。 但現場 AGV 走曲線 + 加減速後，問題變成：

```text
原本：
CP A -> CP B
佔用 Lane(A,B) 一段時間

現場：
CP A -> 曲線軌跡 -> CP B
速度從 0 / v_in 加速
中間可能巡航 / 限速 / 轉彎降速
接近 B 減速
到 B 可能停車、對角度、頂升、取放貨
```

所以 `Lane` 不能只是一條抽象邊。它要變成：

```text
Lane = 拓撲邊 + 幾何曲線 + 運動約束 + 安全包絡 + 時間參數化
```

## 最重要的新概念：Trajectory Envelope

我建議你 v3.5 新增一個核心概念：**Trajectory Envelope，軌跡包絡。**

不要只預約「點」和「邊」，而是預約「車體沿曲線掃過的空間 + 對應時間」。

```csharp
public sealed record EdgeGeometry(
    string FromSiteId,
    string ToSiteId,
    GeometryKind Kind,       // Straight, Arc, Bezier, Polyline
    IReadOnlyList<Pose2d> ControlPoses,
    double LengthMeters,
    double MinTurnRadiusMeters);

public sealed record MotionProfile(
    double MaxVelocity,
    double MaxAcceleration,
    double MaxDeceleration,
    double MaxAngularVelocity,
    double SafetyMarginMeters);

public sealed record TimedPose(
    double X,
    double Y,
    double Theta,
    double S,                // distance along curve
    long TimeMs,
    double Velocity);

public sealed record TrajectoryEnvelope(
    ResourceRef Lane,
    EdgeGeometry Geometry,
    MotionProfile Motion,
    IReadOnlyList<TimedPose> Samples,
    IReadOnlyList<ResourceOccupation> Occupations);
```

然後 TrafficControl 預約的不是「A-B 這條 lane 從 10 到 11 tick」，而是：

```text
Lane:A-B       [t0, t1)
Junction:B     [t1 - approachBuffer, t1 + exitBuffer)
Zone:Turn-B-C  [t_turn_enter, t_turn_exit)
Block:X        [t0 - safetyBuffer, t1 + safetyBuffer)
```

也就是說，**Lane 還在，但 Lane 的 interval 來自真實軌跡時間，而不是固定 hop。**

## 加速 / 減速會改變 edge cost，這點非常要命

你現在不能再把 edge weight 當固定時間用。因為同一條邊的通行時間取決於：

1. 進入邊時的速度 `v_in`
2. 離開邊時是否要停 `v_out`
3. 邊長 `L`
4. 最大速度 `v_max`
5. 加速度 `a_acc`
6. 減速度 `a_dec`
7. 曲率半徑 / 轉彎限速
8. 是否載貨
9. 地面 / 安全區限速
10. 控制器是否在中間點停車

最簡單的時間估算是梯形速度曲線：

```text
d_acc = (v_peak² - v_in²) / (2 * a_acc)
d_dec = (v_peak² - v_out²) / (2 * a_dec)

如果 d_acc + d_dec <= L：
    有巡航段，速度可達 v_max
否则：
    是三角速度曲线，达不到 v_max，需要解 v_peak
```

這會直接影響 SIPP / CCBS 的 safe interval。以前是：

```text
到達 B 的時間 = 到達 A + 1 tick
```

以後要變成：

```text
到達 B 的時間 = 到達 A + TimeParameterizer(A->B, v_in, v_out, load, curve)
```

這就是你 v3 裡 SIPPwRT / continuous-time 的真正落地方向。

## 但有個坑：中間點到底是「必停點」還是「途經點」？

這個現場一定要搞清楚，否則模型會錯。

對 AGV 控制器來說，拓撲點通常有兩種語義：

```text
Stop Point：必須停，到點後完成姿態 / 任務 / 頂升 / 對接
Via Point：只是路徑經過點，車可以平滑通過，不一定停
```

如果你把每個 CP 都當 Stop Point，模擬會過度保守，吞吐會很差；如果你把所有 CP 都當 Via Point，但現場控制器其實會在某些點減速甚至停，預約時間又會過短，安全性會有問題。

所以 Map domain 要新增：

```csharp
public enum SitePassMode
{
    StopRequired,       // 工位、充電點、等待點、交接點
    YieldPossible,      // 需要時可停車讓路
    PassThrough         // 正常平滑通過
}
```

Lane 也要有：

```csharp
public enum LaneMotionMode
{
    StopAndGo,
    SmoothPassThrough,
    CurveFollow,
    DockingApproach,
    RestrictedSpeed
}
```

這會讓 SwarmRoute 從「路線規劃」變成真正懂 AGV 控制語義的系統。

## 你要小心仙工控制器的「局部規劃」問題

如果你們給仙工控制器的是「目標點」，控制器自己決定中間怎麼走曲線，那 SwarmRoute 就不能精確預約軌跡。這時候有三種處理方式：

### 方案 A：SwarmRoute 發整條拓撲路線，控制器按指定邊走

這是最理想的。SwarmRoute 選 CP 序列和 Lane 序列，控制器負責沿每條邊的幾何曲線跟蹤。這時候你可以準確預約 Lane / Zone / Block。

```text
SwarmRoute:
A -> B -> C -> D

Controller:
按地圖中 A-B、B-C、C-D 的曲線行駛
```

這種模式下，SwarmRoute 是上層交通大腦，仙工控制器是底層運動執行器。

### 方案 B：SwarmRoute 只發下一個點，控制器自己找路

這個比較危險。因為控制器可能選了一條 SwarmRoute 沒預約的局部曲線。

這種情況下，你只能做保守預約：

```text
reserve corridor envelope around all possible local trajectories
```

吞吐會下降，但安全。

### 方案 C：控制器能回報 planned path / local trajectory

這也很好。SwarmRoute 發目標點，控制器回傳實際預計曲線，SwarmRoute 再把那條曲線轉成 reservation。

```text
Target point -> controller planned curve -> SwarmRoute validates / reserves -> authorize motion
```

這種方式最貼近真機，但對控制器 API 依賴比較強。

## 我會怎麼改你的架構

### 1. Map Context：從 RoadmapGraph 升級成 Motion Roadmap

Map 仍然負責靜態拓撲，但要多存幾何和運動限制。

```text
Map
- ControlPoint
  - pose: x, y, theta
  - passMode: StopRequired / YieldPossible / PassThrough
  - footprint inflation
  - station / buffer / charger metadata

- Lane
  - from, to
  - geometry: straight / arc / bezier / polyline
  - length
  - width
  - speedLimit
  - accelerationLimit
  - decelerationLimit
  - turnRadius
  - bidirectional physical pair
  - interference zones

- Zone / Block
  - polygon
  - capacity
  - mutex group
```

你現在 Map 已經有 CP、directed lane、mutual-exclusion block、interference set 這些概念。 所以不是推倒重來，而是把 Lane 從「圖邊」補成「可執行曲線」。

---

### 2. PathPlanning Context：新增 Time Parameterizer

PathPlanning 不只找 CP 序列，還要把 CP 序列變成連續時間軌跡。

```text
IPathPlanner
    ↓
Route: A -> B -> C -> D
    ↓
ITrajectoryParameterizer
    ↓
TimedTrajectory + Occupations
```

新增接口：

```csharp
public interface ITrajectoryParameterizer
{
    TimedTrajectory Parameterize(
        RoadmapGraph graph,
        IReadOnlyList<string> siteSequence,
        AgentKinematicProfile agent,
        long releaseTimeMs,
        IReservationView reservations);
}
```

它負責：

```text
- 計算每條 lane 的進入 / 離開時間
- 計算 CP dwell / junction 占用時間
- 根據曲率限速
- 根據是否 stop point 決定 v_out
- 根據載貨狀態調整速度 / 加速度
- 生成 ResourceOccupation
```

v3 的 CCBS / SIPPwRT 最好就基於這個，而不是繼續基於抽象 hop。

---

### 3. TrafficControl Context：從 interval lease 升級成 envelope lease

目前 TrafficControl 擁有 reservation table，負責 interval lease、FreeIntervals、IsFree，這個方向是對的。 但 lease 要變得更豐富。

原來：

```csharp
ResourceLease(ResourceRef resource, TimeInterval interval, string agentId)
```

升級成：

```csharp
ResourceLease(
    ResourceRef resource,
    TimeInterval interval,
    string agentId,
    OccupationKind kind,
    string? trajectoryId,
    double? sStart,
    double? sEnd,
    SafetyEnvelope envelope)
```

其中 `SafetyEnvelope` 可以先簡化，不一定一開始就做完整幾何碰撞。實用路線是三階段：

```text
階段 1：整條 Lane 作為一個互斥資源
階段 2：Lane 切成多個 conflict zone / segment
階段 3：真正的 swept volume 幾何相交檢測
```

我建議你先做階段 1 + 2，會非常實用。

---

### 4. Simulation Context：從 tick executor 升級成 continuous executor

你目前 simulation 用 ManualFleetClock，把 reservation time axis 和 execution tick 對齊，這是 v1 很關鍵的設計。 但真機對齊的是「毫秒 + 位姿 + 速度」，不是「一 tick 一 CP」。

後面 simulator 要支持：

```text
time step = 50ms / 100ms
agent pose = x, y, theta, v
trajectory progress = s
controller state = moving / paused / blocked / docking / error
```

釋放資源也不要按 CP hop，而要按進度：

```text
當車尾離開某個 conflict zone + safety margin
才 release 該 zone
```

這很重要。因為曲線轉彎時，車頭進入下一區域，不代表車尾已經離開上一區域。

## 最關鍵的實作：SEER Controller Adapter

我建議單獨做一個 bounded context 或 infra adapter：

```text
ControllerIntegration/
  SwarmRoute.Controller.Seer
```

它不要污染核心 domain，只做適配。

職責：

```text
1. Map Import
   從仙工地圖 / 點拓撲 / 路徑資料轉成 SwarmRoute RoadmapGraph + MotionRoadmap

2. Route Dispatch
   把 SwarmRoute 的路線轉成仙工控制器能接受的任務 / 點序列 / 路徑指令

3. Telemetry
   讀取車的位置、速度、角度、當前任務、目標點、錯誤碼、是否阻擋

4. Progress Tracking
   判斷車現在在哪條 Lane、s 進度多少、是否偏離預約 envelope

5. Lease Release
   根據實際進度釋放已通過資源

6. Stall Handling
   若車停住超過 N 秒，保留當前安全區，觸發 replan / reroute / 人工介入

7. ETA Correction
   用真實執行時間校準 edge travel time
```

接口可以這樣設：

```csharp
public interface IRobotControllerAdapter
{
    Task<RobotState> GetStateAsync(string robotId, CancellationToken ct);

    Task<ControllerRouteHandle> DispatchRouteAsync(
        string robotId,
        ControllerRoute route,
        CancellationToken ct);

    Task PauseAsync(string robotId, PauseReason reason, CancellationToken ct);

    Task ResumeAsync(string robotId, CancellationToken ct);

    IAsyncEnumerable<RobotTelemetry> SubscribeTelemetryAsync(
        string robotId,
        CancellationToken ct);
}
```

不要把仙工 API 細節塞進 PathPlanning / TrafficControl。核心保持乾淨，仙工只是 adapter。

## 現場最該先做的一個實驗

你們現在可以先做一個很小但價值極高的測試：**測出控制器實際運動模型。**

選 5 條典型路段：

```text
1. 短直線
2. 長直線
3. 90 度轉彎
4. S 曲線 / 弧線
5. 工位進出 / 狹窄區
```

每條路跑 20 次，記錄：

```text
time, x, y, theta, velocity, currentPoint, targetPoint, taskState
```

然後得到：

```text
- 實際加速度
- 實際減速度
- 最大速度
- 曲線降速規則
- 到點是否停車
- 中間點是否平滑通過
- 同一路段 P50 / P95 / P99 通行時間
- 實際軌跡相對地圖曲線的最大偏差
```

最後把結果塞回 SwarmRoute：

```text
planner cost 用 P50 / P75
reservation interval 用 P95 / P99 + safety buffer
```

這會非常接地氣。因為你不需要完全知道仙工控制器內部怎麼算，只要用現場數據把它當黑盒建模。

## 你後面真正的安全邊界

這句很重要：

> **SwarmRoute 的安全保證，只能建立在「控制器實際行駛不超出 SwarmRoute 預約的 envelope」之上。**

如果控制器會為了避障自己繞一下，而且繞出原本 Lane / Zone，那 SwarmRoute 的 reservation 就失效了。這不是 SwarmRoute 的錯，而是責任邊界沒定義清楚。

所以要定三條規則：

```text
1. 正常行駛時，控制器必須被限制在指定 Lane / corridor envelope 內。
2. 如果局部避障需要離開 envelope，控制器必須停車或請求重新授權。
3. SwarmRoute 根據 telemetry 發現偏離 envelope，要立刻擴大保護區 / 暫停相關車 / 觸發 replan。
```

這樣才能把上層 MAPF 的安全性和底層控制器的真實行為接起來。

## 最後的判斷

先生，你現在這個補充讓方向更清楚了：

**SwarmRoute 不應該嘗試取代仙工控制器。**
仙工控制器負責底層定位、導航、曲線跟蹤、加減速、車體控制。
**SwarmRoute 應該做仙工控制器上面的 traffic brain：決定哪台車、何時、被允許佔用哪條曲線走廊 / 哪個交叉區 / 哪個工位入口。**

所以 v3 後最有價值的不是再堆 solver，而是先做：

```text
點拓撲 MAPF
    ↓
曲線幾何
    ↓
加減速時間模型
    ↓
車體掃掠包絡
    ↓
連續時間 reservation
    ↓
仙工控制器 adapter
```

這一步做完，你的項目就從「MAPF 演算法系統」變成了「能接真實 AGV 控制器的交通控制核心」。这比单纯多实现一个算法值钱得多。
